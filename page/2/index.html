分页和分段的区别

  • 页是信息的物理单位，分页是为了实现离散分配方式、消减内存零头、提高内存利用率。仅仅处于系统内存管理的需求而非用户需求；段是信息的逻辑单位，分段目的是为了更好满足用户需求。
  • 页的大小固定，由系统把逻辑地址分为页号和页内地址两部分；而段是由用户编写程序决定，长度不固定

进程与线程的区别、应用场景

  • 线程：CPU调度的基本单位，是进程内部的一个执行流，在进程的地址空间内运行。

  • 进程：系统分配资源的基本单位、能独立运行的基本单位，是拥有一个/多个执行流的线程组。

  • 创建、销毁一个线程相对于进程的成本要低（创建进程要创建PCB、虚拟地址空间、页表、维护映射、加载数据到内存等；而创建线程只需要一个PCB指向进程的虚拟空间即可，同理销毁线程只需要销毁PCB）。

  • 线程的切换比进程的切换容易的多，进程切换需要PCB切换、页表切换；线程切换需要PCB切换，页表不切换。

  • 进程拥有自己独立的虚拟地址空间，而一个进程中的多个线程共享进程的虚拟地址空间。因此线程缺乏访问控制，进程中的一个线程出错，会终止掉整个进程，从而导致其他线程也凉凉，而一个进程出错，不会影响另一个进程。

  • 线程共享进程空间，通信成本容易；进程通信难，成本高。

  • 线程的应用场景：等待慢速I/O时，交给一个线程等待，接着做其他事情；

  • 进程的应用场景：有安全、稳定的需求时；需要速度时；需要速度和安全时；

多线程与多进程：

对比维度 多进程 多线程 总结
数据共享、同步 数据共享复杂，需要用IPC（进程间通信，InterProcess Communication）；数据是分开的，同步简单 因为共享进程数据，数据共享简单，但也是因为这个原因导致同步复杂 各有优势
内存、CPU 占用内存多，切换复杂，CPU利用率低 占用内存少，切换简单，CPU利用率高 线程占优
创建销毁、切换 创建销毁、切换复杂，速度慢 创建销毁、切换简单，速度很快 线程占优
编程、调试 编程简单，调试简单 编程复杂，调试复杂 进程占优
可靠性 进程间不会互相影响 一个线程挂掉将导致整个进程挂掉 进程占优
分布式 适应于多核、多机分布式；如果一台机器不够，扩展到多台机器比较简单 适应于多核分布式 进程占优

进程间通信（IPC）

主要方式包括：管道、FIFO、消息队列、信号量、共享内存、Socket。

  • 管道（无名管道）：半双工，只能在具有公共祖先额度两个进程中使用（通常为父进程与子进程）。

  • FIFO（命名管道）：可以在不想关的进程间交换数据。FIFO是一种文件类型。 shell命令使用FIFO将数据从一条管道传送到另一条管道时，无须创建中间的临时文件。

    无名管道与有名管道的区别在于单向通信以及有关联的进程。

  • 消息队列：消息的链接表，存储在内核中，有消息队列标识符标识。并不一定以先进先出的顺序取消息，可以按照消息的类型字段取消息。

  • 信号量：与管道，FIFO以及消息队列不同。它是一个计数器，用于为多个进程提供共享数据对象的访问

  • 共享内存：允许两个或多个进程共享一个给定的存储区，因为数据不需要在客户进程和服务器进程之间复制，所以这是最快的IPC。信号量用于同步共享存储访问。

  • socket：网络中进程之间通信。open—write/read—close”模式的一种实现。

动态链接dll和静态链接lib：

这两者都是共享代码的一种方式。共享代码就是说，一帮程序员要开发一个项目，如果每个人做的不同块中有相同的功能，那么，在最后的封装中，可以把相同代码塞到链接库里（这样就不用写n遍了，节约空间）。

  • 静态链接：

    优点：

    1. 代码装载速度快，执行速度略快。原因：你一开始就调用了整个静态链接库，so，不用一个个装载当前用的函数或者方法；

      1. 只需保证在开发者的计算机中有正确的.LIB文件，在以二进制形式发布程序时不需考虑在 用户的计算机上.LIB文件是否存在及版本问题，可避免DLL地狱等问题。

    缺点：

    1. 使用静态链接生成的可执行文件体积较大，包含相同的公共代码，造成浪费（这就是重复代码一遍遍被封装的后果）。
    2. .lib用空间换时间。

  • 动态链接：

    优点：

    1. 更加节省内存并减少页面交换。原因：仅仅调用自己需要的，不需要的就不调用，所以减少内存（减少页面交换同理）。
    2. DLL文件与EXE文件独立，只要输出接口不变（即名称、参数、返回值类型和调用约定不变），更换DLL文件不会对EXE文件造成任何影响，因而极大地提高了可维护性和可扩展性。
    3. 不同编程语言编写的程序只要按照函数调用约定就可以调用同一个DLL函数（当然你必须遵守函数调用约定。这东西可以视为一种协议。）
    4. 由于具有以上三种性质，所以.dll适用于大规模的软件开发，使开发过程独立、耦合度小，便于不同开发者和开发组织之间进行开发和测试（你可以理解为模块化程度高）。

    缺点：

    1. 使用动态链接库的应用程序不是自完备的，它依赖的DLL模块也要存在。
    • 如果使用载入时动态链接，程序启动时发现DLL不存在，系统将终止程序并给出错误信息。
    • 使用运行时动态链接，系统不会终止，但由于DLL中的导出函数不可用，程序会加载失败。
    1. 速度比静态链接慢。
    2. 如果新模块与旧的模块不兼容，那么那些需要该模块才能运行的软件，统统撕掉。这在早期Windows中很常见（譬如 “缺少XXX.dll，程序无法运行”）。
    3. 用时间换空间。

Maven自动化部署

一般的项目开发部署过程：

  • 每个团队将所有的项目代码提交到 SVN 或者代码库中并打上标签。
  • 从 SVN 上下载完整的源代码。
  • 构建应用。
  • 存储构建输出的 WAR 或者 EAR 文件到一个常用的网络位置下。
  • 从网络上获取文件并且部署文件到生产站点上。
  • 更新文档并且更新应用的版本号。

其中涉及多个团队，很容易出错，maven可以结合如下方案实现自动化部署：

  • 使用 Maven 构建和发布项目
  • 使用 SubVersion， 源码仓库来管理源代码
  • 使用远程仓库管理软件（Jfrog或者Nexus） 来管理项目二进制文件。

具体使用可以查看菜鸟教程Maven自动化部署。

Solr索引

Solr索引可以接受来自许多不同来源的数据，包括XML文件，逗号分隔值（CSV）文件，从数据库中的表中提取的数据以及常用文件格式（如Microsoft Word或PDF）中的文件。

以下是将数据加载到Solr索引的三种最常用方法：

  • 使用基于Apache Tika构建的Solr Cell框架来提取二进制文件或结构化文件，如Office，Word，PDF和其他专有格式。
  • 通过从可以生成此类请求的任何环境向Solr服务器发送HTTP请求来上载XML文件。
  • 编写自定义Java应用程序以通过Solr的Java客户端API（在客户端API中更详细地描述）来提取数据。

Solr引擎

一个基于Apache Lucene的开源的快速的企业搜索引擎。

Solr将数据组织在文档document和字段field里面；schema.xml是用户手动编辑的模式文件。

Tips: 字符串solr是org.apache.solr.schema或org.apache.solr.analysis的缩写

以下部分描述了Solr如何分解并处理文本数据。有三个主要概念需要理解：分析器Analuzer，标记器Tokenizer和过滤器Filter。

  • 分析器Analyzer

    分析器在输入阶段（建索引阶段）和检索阶段使用。分析器可以是单个类，也可以由一系列标记器和过滤器类组成。

    Analyzer检查输入文本，并生成token流。

    Analyzer是schema.xml中<filedType>元素的子元素，在同目录的solrconfig.xml中定义。一般而言，只有solr.TextField或solr.SortableTextField类型的字段可以定义一个Analyzer。

  • Tokenizer

    Tokenizer的工作是将文本流分解为标记，其中每个标记（通常）是文本中字符的子序列。

    使用Tokenizer可以丢弃输入流中的字符，例如空格或其他分隔符。它们也可以被添加或替换，例如将别名或缩写映射到规范化形式。除了文本值之外，Token还包含各种元数据，例如Token在字段中出现的位置。

  • Filter

    Filter检查，过滤Token流。

    Filter的工作比Tokenizer简单，与其类似。Filter的输入是不同的Token流，它的作用是选择更改或者舍弃某些token，如将drove改为drive，将Run改为run。

Maven学习过程（一）

Maven 提倡使用一个共同的标准目录结构，Maven 使用约定优于配置的原则，大家尽可能的遵守这样的目录结构。如下所示：

目录 目的
${basedir} 存放pom.xml和所有的子目录
${basedir}/src/main/java 项目的java源代码
${basedir}/src/main/resources 项目的资源，比如说property文件，springmvc.xml
${basedir}/src/test/java 项目的测试类，比如说Junit代码
${basedir}/src/test/resources 测试用的资源
${basedir}/src/main/webapp/WEB-INF web应用文件目录，web项目的信息，比如存放web.xml、本地图片、jsp视图页面
${basedir}/target 打包输出目录
${basedir}/target/classes 编译输出目录
${basedir}/target/test-classes 测试编译输出目录
Test.java Maven只会自动运行符合该命名规则的测试类
~/.m2/repository Maven默认的本地仓库目录位置

pom.xml

POM( Project Object Model，项目对象模型 ) 是 Maven 工程的基本工作单元，是一个XML文件，包含了项目的基本信息，用于描述项目如何构建，声明项目依赖，等等。

执行任务或目标时，Maven 会在当前目录中查找 POM。它读取 POM，获取所需的配置信息，然后执行目标。

POM 中可以指定以下配置：

  • 项目依赖 dependencies
  • 插件 plugins
  • 执行目标 goals
  • 项目构建 profile
  • 项目版本 project version
  • 项目开发者列表 developers
  • 相关邮件列表信息 mailing list

所有 POM 文件都需要 根节点project 元素和三个必需字段：groupId，artifactId，version。
